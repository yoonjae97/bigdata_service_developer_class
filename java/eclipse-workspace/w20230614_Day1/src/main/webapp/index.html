Get과 Post 방식(method)

파라미터를 url 인코딩으로 변경해서 전달한다()

  1. Get : a태그로 파라미터를 요청하는 방식
    • 링크 : 다른 사이트로 이동 (절대주소 https://)
    • 네비게이션 : 내 사이트 내에서 다른 사이트로 이동 (상대주소 /, ./)
    • 주소 공유
    • 검색
    • **파라미터가 주소의 일부
    • 파라미터를 ? 이하의 쿼리스트링으로 포함한다. 브라우저마다 다른 길이 제한이 있다.
  2. Post : 양식(form) 제출로 파라미터를 요청하는 방식
    • 로그인
    • 회원가입
    • 결제
    • 게시판 생성, 수정
    • 파일 업로드
    • **파라미터가 동적페이지에서 처리할 대상 (url에 파라미터 미포함)
    • 요청헤더본문에 쿼리스트링으로 파라미터를 포함한다. (파라미터의 길이제한이 없다, Get방식보다 통신이 조금 느리다, 보안이 강화된다.)

http 통신과 status 상태

  • status 100 : http로 소켓통신을 할 때 (메세지를 주고 받을 때)
  • status 200 : 성공
  • status 300 : 리다이렉트 상태 (서버에서 서버의 리소스를 요청, 로그인을 요청 -> 로그인 처리 -> (index, login) 302)
  • status 400 : 클라이언트가 잘못 요청했을 때
    • 404 : 해당 page의 리소스가 존재하지 않음
    • 405 : get방식으로 요청해야 하는데 post로 요청했거나 반대인 경우
  • status 500 : 개발자가 예외처리를 하지 않을 때 or 서버오류가 있을 때

url 구조와 리소스

서버에 저장된 리소스(공유될 수 있는 자료)를 요청하는데 사용된다.

https://naver.com/book/detail.do?bookId=123123

  • https : 통신규약 http 통신을 하겠다
  • naver.com : 서버주소 (도메인 : ip를 랩핑하는 것)
  • /book/detail.do : 서버에서 리소스의 주소 (path)
  • ?bookId=123123 : 쿼리스트링 파라미터를 포함하는 문자열
    • 동적리소스에서 처리해야할 내역을 전달 (123123 아이디의 책을 찾아서 detail.do에 출력)
    • url 인코딩으로 파라미터 값을 처리한다.
    • 값이 너무 긴 것을 전달할 수 없다.

동적, 정적 리소스

정적 리소스

요청하는 리소스가 변화가 없는 것 img, 동영상, html, css, js ... (웹서버(아파치)가 처리하는 리소스)

동적 리소스

요청하는 리소스가 실행되어 변하는 것 (db 접속 (jdbc)해서 조회 수정 삭제 등록을 하는 것42

  • java servlet (컴파일), jsp JavaServletPage (인터프리터 엔진)
  • php, asp
  • python django
  • node.js(v8)

CRUD Create Read Update Delete

  • Read : 리스트(페이징, 정렬, 검색), 상세, 수정양식
  • Create : 등록 액션
  • Update : 수정 액션
  • Delete : 삭제 액션
